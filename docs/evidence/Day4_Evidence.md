## 2. 发布类型与字段设计

| 发布类型 | 对应数据集合 | 关键字段 | 设计理由 |
|---------|------------|---------|---------|
| 二手交易 | trades | title, category, price, condition, location, description | 分类帮助买家筛选，价格是核心决策因素，成色反映商品新旧 |
| 失物招领 | lostFounds | title, type, itemName, location, eventTime, description | type 区分寻物/招领，itemName 精确描述物品，eventTime 定位时间 |
| 拼单搭子 | groupBuys | title, type, targetCount, deadline, location, description | targetCount 定义人数目标，deadline 明确截止时间，type 分类 |
| 跑腿委托 | errands | title, taskType, reward, from, to, deadline, description | reward 展示酬劳，from/to 清晰定义取送路径，taskType 区分任务 |

所有类型的公共字段：标题、地点、描述。这三者适用于任何发布场景。

## 3. 表单校验规则

| 校验项 | 规则 | 适用类型 |
|-------|------|---------|
| 标题 | 不能为空 | 全部 |
| 地点 | 不能为空 | 全部 |
| 描述 | 不能为空 | 全部 |
| 商品分类 | 不能为空 | 二手交易 |
| 价格 | 必须大于 0 | 二手交易 |
| 成色 | 必须选择 | 二手交易 |
| 物品名称 | 不能为空 | 失物招领 |
| 发生时间 | 必须选择 | 失物招领 |
| 拼单类型 | 不能为空 | 拼单搭子 |
| 目标人数 | 不能少于 2 人 | 拼单搭子 |
| 截止时间 | 必须选择 | 拼单搭子 + 跑腿委托 |
| 任务类型 | 不能为空 | 跑腿委托 |
| 酬劳 | 不能为负数 | 跑腿委托 |
| 校区 | 必须选择 | 二手交易 |
| 取件地点 | 不能为空 | 跑腿委托 |
| 送达地点 | 不能为空 | 跑腿委托 |
| 图片地址 | 可选 | 全部 |

> 注：校区仅在二手交易时为必填，图片地址为通用可选字段。

校验在 `validateForm()` 中按 `publishType` 分类执行，错误存入 `errors` 对象并通过 FormField 组件的 error prop 显示。

## 8. 今日反思

发布表单、表单校验和数据新增是前端应用从"只读展示"走向"用户交互"的三要素。

发布表单的本质是将用户意图转化为结构化数据——用户想卖一本教材，表单将"教材""15元""八成新""图书馆"这些信息组织成 `{title, price, condition, location}` 的结构化对象。

表单校验的本质是在数据进入系统之前设置质量关卡——如果用户没填标题就提交，列表页就会出现没有标题的空卡片；如果价格填了负数，后续的排序和统计就会出错。

数据新增的本质是前端不再是孤立的展示层——通过 POST 请求，前端与 JSON Server 形成了完整的读写闭环。这个闭环是理解前后端分离开发的基础。

三者共同完成了"用户输入 → 前端校验 → 接口提交 → 数据写入 → 页面刷新"的完整链路。这个链路在 Day5 的个人中心和收藏功能中会被反复复用。
